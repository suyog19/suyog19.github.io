# Software Signal learner experience matrix

Authoritative frontend remediation specification for #223 and epic #222. Backend contracts and state machines remain authoritative and unchanged. Product Owner approval: pending explicit review.

## Route inventory

Public: `/training/`, both `/training/<course>/` routes, `/training/policies/`, `/training/provider/`. Authentication/application: `/learn/`, `/apply/` including valid, missing and stale correction context. Private: `/my-learning/`, `/my-learning/payment/`, `/my-learning/change/`, `/my-learning/balance/`. Cross-journey: `/contact/`, protected-route authentication return and approved email deep links.

## Universal experience rules

Every state leads with status, meaning, next action, then deadline/update. At most one action is visually dominant. References, detailed financial history and policy versions use `View details`. Authentication preserves a validated safe destination and never puts answers, email, IDs or tokens into `continue`. A failed or mismatched fetch clears private content and stale financial actions. Dates come only from learner-safe fields; unavailable dates use “We will email you when a meaningful update is recorded.” Support context contains a category and safe summary, never sensitive identifiers. Dynamic errors focus the summary/field; status updates use a polite live region. At 360px status, date and action remain in reading order with no horizontal overflow.

Legend: `A` = backend action/status code; `D` = backend date field; `No change` = no backend work. Validation always includes unit fixture, desktop, 390px, keyboard and stale/refresh recovery; High/Critical also require indexed evidence.
